# streetlight-backend 改造说明

> 本文档说明本目录（`streetlight-backend`）相对于参考 demo `参考资料mqtt/mqtt01-master` 做了哪些改造、为什么这么改，以及怎么联调运行。
> 
> 适配目标：[`智慧路灯_接口规范.md`](../智慧路灯_接口规范.md) §3–§7（13 个 REST 接口 + WebSocket 实时通道 + 自动联动/离线检测业务规则）。
> 硬件参考：[`参考资料mqtt/D9_iot_cloud_oc_light`](../参考资料mqtt/D9_iot_cloud_oc_light/D9_iot_cloud_oc_light/README.md)（BearPi-HM_Nano 智能台灯，已通过 MQTT 接华为云 IoTDA）。

---

## 1. 背景

### 1.1 起点和目标的错位

参考 demo `mqtt01-master` 的形态：

| 项        | demo 现状                                                             |
| -------- | ------------------------------------------------------------------- |
| 角色       | Spring Boot 单服务，演示 MQTT 收发的最小可运行例子                                  |
| Broker   | `tcp://43.136.185.53:1883`（demo 作者自己的测试 EMQX，**外网不一定还在**）           |
| 订阅 topic | `leddata`（一个写死的 topic）                                              |
| Payload  | `{ "toilet_state": "...", "light_state": "..." }`（智能马桶演示用）          |
| 发布 topic | `abc001`（演示用）                                                       |
| 落库       | `Db.use().execute("insert into d_record(data1, data2) ...")`（无业务模型） |
| REST 接口  | `GET /publishTopic` / `/on` / `/off`（3 个演示 GET）                     |
| 鉴权       | 无                                                                   |
| 实时推送     | 无                                                                   |

而我们的目标是：

- 硬件已经按 D9_iot_cloud_oc_light 接入**华为云 IoTDA**，走 `iot-mqtts.cn-north-4.myhuaweicloud.com`，不是 EMQX。
- 数据按华为云产品模型上报：`service_id=Light`，属性 `Luminance`(int) + `LightStatus`("ON"/"OFF")；命令 `Light_Control_Led`，参数 `Led=ON/OFF`。
- 前端 `产品原型-优化/` 需要 13 个 REST 接口 + WebSocket 实时通道，所有字段、状态机已在接口规范里钉死。

所以 `mqtt01-master` 几乎只剩"Spring Boot 工程骨架 + Paho MQTT 写法"可以复用，业务层全部需要补。

### 1.2 改造原则

1. **保留 demo 的工程结构**（Maven + Spring Boot 2.3.5 + JDK 8 + Hutool 5.x），降低同学的上手成本。
2. **不强绑华为云**：通过配置项 `streetlight.mode = huawei | emqx | both` 切换接入方式，方便在没有真硬件时用 paho-mqtt 脚本模拟设备调试。
3. **业务层与接入层解耦**：所有上报都收敛到 `DeviceService.onTelemetry(deviceId, light, lamp, ts)`，下行命令通过 `CommandSender` 接口抽象（华为云 HTTPS / EMQX MQTT 两种实现）。

后端在中间承担四个角色：

1. **华为云推送接收器**：在 IoTDA 控制台配置"规则 → 数据转发 → HTTP 推送"指向 `POST /api/huawei/push`，落地到 `HuaweiPushController`。
2. **华为云命令下发器**：`HuaweiCommandSender` 同步调 `POST /v5/iot/{project_id}/devices/{device_id}/commands`，IAM token 在 `HuaweiIamTokenHolder` 里自动刷新。
3. **EMQX 订阅 + 发布**（可选）：`MqttAcceptCallback` / `MqttSendClient` / `MqttCommandSender`，topic 前缀 `group8_streetlight`，仅 `mqtt.enabled=true` 时启用。
4. **WebSocket 推送 + REST 接口**：所有 `device-update` / `alarm-new` / `lamp-feedback` / `heartbeat` 事件经 `WsServer#broadcast` 推给已握手成功的浏览器；REST 接口集中在 `controller/` 包。

## 3. 七大改造点（对照接口规范 §9.2）

### 3.1 新增 WebSocket 端点

**为什么**：设备光照、灯状态、心跳这些数据是流式的，REST 轮询体验差且浪费。接口规范 §4 决定走"后端 WebSocket 中转"。

**做了什么**：

- `pom.xml` 加依赖：
  
  ```xml
  <dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-websocket</artifactId>
  </dependency>
  ```

- 新增 `config/WebSocketConfig.java` 注册 `/ws` 端点 + 跨域允许 + 握手拦截器。

- 新增 `ws/WsServer.java`（继承 `TextWebSocketHandler`）：维护 `Map<sid, WebSocketSession>`，对外暴露 `broadcast(type, payload)`；30s 一次服务端 `heartbeat`（用 `@Scheduled` 实现，与接口规范 §4.2 一致）。

- 新增 `ws/WsHandshakeInterceptor.java`：从 URL `?token=...` 取出 JWT，校验通过才放行握手；不通过返回 401。

- 业务侧通过 `@Autowired private WsServer ws;` 注入，调用 `ws.broadcast("device-update", deviceObj)` 即可推给所有在线浏览器。

事件类型与接口规范 §4.2 表完全一致：`device-update` / `alarm-new` / `lamp-feedback` / `heartbeat`。

### 3.2 Topic 命名约定改造（仅 EMQX 通道）

**为什么**：班级 EMQX broker 是公开的，所有组共用；不加前缀会与其他组互相串数据。接口规范 §5 强制 `group8_streetlight/<id>/<kind>` 前缀。

**做了什么**：

- `mqtt01-master` 里硬编码 `mqttAcceptClient.subscribe("leddata", 0)` 和 `MqttSendClient.publish(false, "abc001", ...)`。

- 本工程在 `application.yml` 抽出 `mqtt.topic-prefix: group8_streetlight`，由 `MqttProperties` 注入。

- `MqttAcceptCallback#connectComplete` 改为按前缀订阅三个 topic 模式：
  
  ```java
  String prefix = mqttProps.getTopicPrefix();
  mqttAcceptClient.subscribe(prefix + "/+/data", qos);
  mqttAcceptClient.subscribe(prefix + "/+/heartbeat", qos);
  mqttAcceptClient.subscribe(prefix + "/+/feedback", qos);
  ```

- `MqttSendClient#publishCommand(deviceId, action)` 自动拼成 `group8_streetlight/<id>/cmd`。

### 3.3 Payload 解析重写

**为什么**：demo 解析的是智能马桶的 `toilet_state` / `light_state`，与本项目设备模型完全不同。

**做了什么**：`MqttAcceptCallback#messageArrived` 完全重写——

```java
Matcher m = TOPIC_PATTERN.matcher(topic);  // ^([^/]+)/([^/]+)/(data|heartbeat|feedback)$
String deviceId = m.group(2);
String kind = m.group(3);
switch (kind) {
    case "data":      deviceService.onTelemetry(deviceId, json.getInt("light"), json.getBool("lamp"), ts); break;
    case "heartbeat": deviceService.onTelemetry(deviceId, null, null, ts); break;
    case "feedback":  deviceService.onCommandFeedback(deviceId, Boolean.TRUE.equals(json.getBool("ok")), json.getStr("action")); break;
}
```

字段与接口规范 §5 的 topic 表完全对齐。

### 3.4 Broker 地址 + MySQL 接入

**为什么**：demo 的 `tcp://43.136.185.53:1883` 是 demo 作者自己的；硬件实际接的是华为云 IoTDA；DB 也要换到班级的 `dream22`。

**做了什么**：

- `application.yml` 重写：
  
  - `spring.datasource.url = jdbc:mysql://47.108.58.107:3306/dream22?...`，账号 `root` / `c0765083cd3f57ab`（来自 `总览信息.md`）。
  - 新增 `streetlight.mode = huawei`（生产）；`emqx` 用于无硬件时本地模拟；`both` 兼容。
  - 新增 `huawei.iot.*` 块：endpoint / project-id / IAM 凭据 / `service-id=Light` / `command-name=Light_Control_Led` / `report-service-id=Light`（这些字段名与 D9_iot_cloud_oc_light 产品模型一一对齐，见 §6）。
  - `mqtt.host-url = tcp://47.108.58.107:1883`（班级 EMQX，仅当 `mode != huawei` 时启用）。

- Hutool 的 `Db.use()` 默认会读 Spring 注入的 `DataSource`，但需要一个桥接 Bean——见 `config/HutoolDbBridge.java`：
  
  ```java
  @PostConstruct
  public void register() { DSFactory.setCurrentDSFactory(new SimpleDataSourceFactory(dataSource)); }
  ```

- MQTT 相关 Bean 全部加 `@Conditional(MqttCondition.class)`，根据 `mqtt.enabled=true/false` 决定是否注入；同样 `HuaweiCommandSender` 加 `@ConditionalOnExpression("streetlight.mode=huawei||both")`，避免 `mode=emqx` 时还想去拿 IAM token。

### 3.5 数据库建表

**为什么**：demo 的 `d_record(data1, data2)` 是玩具表；按接口规范 §7.1 的 DDL 5 张业务表才能放下设备 + 历史 + 告警 + 事件 + 用户。

**做了什么**：

- `resources/sql/schema.sql` 完整收录了接口规范 §7.1 的 DDL（`users` / `devices` / `device_status` / `history` / `alarms`）+ 3 台演示设备的 `INSERT IGNORE`。
- 在 §7.1 DDL 基础上，`devices` 表多加了一列 `huawei_device_id VARCHAR(64)`：本项目业务 ID（如 `light001`）与华为云 IoTDA 内部设备 ID 不一样，需要一张映射；下行命令时拿这个字段拼 URL。
- `config/DataInitializer.java`（`CommandLineRunner`）启动时自动跑 `schema.sql`（`IF NOT EXISTS`，幂等），再把 `application.yml` 里 `streetlight.default-accounts` 写进 `users` 表（密码 BCrypt 哈希）。第一次启动就有 `admin/admin123` 和 `municipal01/123456` 可用。

### 3.6 完整实现 13 个 REST 接口

**为什么**：demo 只有 `GET /publishTopic` / `/on` / `/off` 3 个演示 GET，远远不够。

**做了什么**：

| 接口规范 § | HTTP   | 路径                          | 实现位置                                  |
| ------ | ------ | --------------------------- | ------------------------------------- |
| §3.1   | POST   | `/api/login`                | `AuthController#login`                |
| §3.2   | GET    | `/api/devices`              | `DeviceController#list`               |
| §3.2   | GET    | `/api/devices/{id}`         | `DeviceController#get`                |
| §3.2   | POST   | `/api/devices`              | `DeviceController#add` (admin)        |
| §3.2   | DELETE | `/api/devices/{id}`         | `DeviceController#remove` (admin)     |
| §3.3   | GET    | `/api/devices/{id}/history` | `DeviceController#history`            |
| §3.4   | POST   | `/api/devices/{id}/cmd`     | `DeviceController#sendCmd`            |
| §3.4   | PUT    | `/api/devices/{id}/policy`  | `DeviceController#savePolicy`         |
| §3.5   | GET    | `/api/alarms`               | `AlarmController#list`                |
| §3.5   | POST   | `/api/alarms/{id}/resolve`  | `AlarmController#resolve`             |
| §3.6   | GET    | `/api/events`               | `EventController#list`                |
| §3.7   | POST   | `/api/assistant`            | `AssistantController#ask`（占位回答）       |
| §4     | WS     | `/ws?token=...`             | `WsServer` + `WsHandshakeInterceptor` |

**鉴权**：`auth/AuthFilter`（Servlet `Filter`，`@Order(1)`）拦截所有 `/api/**`，放行 `/api/login` 和 `/api/huawei/push`；其余请求要求 `Authorization: Bearer <jwt>`；admin 专属接口（POST/DELETE devices）按 token 中 `role` 字段二次过滤。JWT 用 Hutool 的 `JWTUtil`，密钥从 `streetlight.jwt.secret` 读，默认 24h 过期。

**错误码**：统一 `model/ApiError { code, message }`；`controller/GlobalExceptionHandler` 把未捕获的异常包成 500。状态码语义与接口规范 §1.4 表对齐（401/403/404/409 各有用到）。

**CORS**：`config/CorsConfig` 全开，方便前端从本地静态服务器（如 VSCode Live Server）跨域调试。生产上接反向代理后可以去掉。

### 3.7 业务逻辑：自动联动 + 离线检测

**为什么**：接口规范 §6 明确要求这两个规则；demo 完全没有业务层概念。

**做了什么**：

#### 自动联动（§6.1）

收敛点：`service/DeviceService#onTelemetry(deviceId, light, lamp, eventTime)`。无论数据是从华为云 push（`HuaweiPushController`）还是 EMQX 订阅（`MqttAcceptCallback`）进来，统一在这里：

1. 写 `device_status`（覆盖式）+ 写一条 `history`；
2. 如果之前 `online=false` 现在恢复，生成"通信恢复"告警；
3. WebSocket 广播 `device-update`；
4. 若 `device.autoMode && device.online && light != null`：
   - `shouldBeOn = light < threshold`
   - 当前 `lamp != shouldBeOn` 时下发 `turn_on` / `turn_off` 给 `CommandSender`，写"自动开灯"/"自动关灯"告警 + 事件，并**提前回填** `device_status.lamp`（避免下次上报还没到时重复触发；真实设备执行后会再 `feedback` 刷新）。

#### 离线检测（§6.2）

`task/OfflineScanner`：`@Scheduled(fixedDelay = ${streetlight.offline-scan-interval-ms:30000})` 每 30s 扫一次：

```sql
SELECT d.id, s.online, s.last_seen
  FROM devices d LEFT JOIN device_status s ON d.id = s.device_id
```

`online=1` 且 `last_seen < now - 60s` → `markOffline`，生成"设备离线"告警（级别"中"），WebSocket 推 `alarm-new` + `device-update`。超时阈值 `streetlight.offline-timeout-sec` 可配。

## 4. 与 D9_iot_cloud_oc_light 设备端的对接

D9 例程使用华为云**产品模型**上报，本工程做了字段级对齐。

### 4.1 上报方向（设备 → 后端）

D9 代码 `iot_cloud_oc_sample.c#deal_report_msg`：

```c
service.service_id = "Light";
luminance.key = "Luminance"; luminance.type = INT;
led.key = "LightStatus";     led.value = g_app_cb.led ? "ON" : "OFF";
```

经华为云 IoTDA 转发后 HTTP 推送 body：

```json
{
  "resource": "device.property", "event": "report",
  "notify_data": {
    "header": { "device_id": "5fxxx_light001", ... },
    "body": { "services": [{
      "service_id": "Light",
      "properties": { "Luminance": 268, "LightStatus": "ON" }
    }]}
  }
}
```

后端解析见 `huawei/HuaweiPushController#handleProperty`：

```java
if (!props.getReportServiceId().equalsIgnoreCase(svc.getStr("service_id"))) continue;
if (p.containsKey("Luminance")) light = ((Number) p.get("Luminance")).intValue();
if (p.containsKey("LightStatus")) lamp = "ON".equalsIgnoreCase(p.getStr("LightStatus"));
deviceService.onTelemetry(deviceId, light, lamp, eventTime);
```

`deviceId` 查表：`devices.huawei_device_id = header.device_id` → 本项目业务 ID。

`device.status` 推送（ONLINE/OFFLINE）也会进 `handleStatus`，OFFLINE 直接 `markOffline`。

### 4.2 下行方向（后端 → 设备）

D9 代码 `deal_cmd_msg` 解析的命令：

```c
if (0 == strcmp(cmdname, "Light_Control_Led")) {
    if (0 == strcmp(led, "ON"))  Light_StatusSet(ON);
    else                          Light_StatusSet(OFF);
}
```

后端 `huawei/HuaweiCommandSender#send` 调华为云 IoTDA 同步命令 API：

```
POST {endpoint}/v5/iot/{project_id}/devices/{device_id}/commands
Header: X-Auth-Token: <iam_token>
{
  "service_id":  "Light",
  "command_name":"Light_Control_Led",
  "paras": { "Led": "ON" | "OFF" }
}
```

`action="turn_on"` → `Led=ON`，反之亦然。

### 4.3 字段映射全表

| 层   | 字段      | 本项目                         | D9 设备端                                             | 华为云上报体                                           |
| --- | ------- | --------------------------- | -------------------------------------------------- | ------------------------------------------------ |
| 光照  | int     | `device.light`              | `report.lum`                                       | `services[].properties.Luminance`                |
| 灯状态 | boolean | `device.lamp`               | `g_app_cb.led`                                     | `services[].properties.LightStatus` ("ON"/"OFF") |
| 命令  | string  | `action`=`turn_on/turn_off` | `command_name=Light_Control_Led, paras.Led=ON/OFF` | 同左                                               |

### 5.1 模式选择速查

| 场景                   | `streetlight.mode` | `mqtt.enabled` | 备注                |
| -------------------- | ------------------ | -------------- | ----------------- |
| 真硬件接华为云（生产）          | `huawei`           | `false`        | 必填 `huawei.iot.*` |
| 没硬件，用 paho-mqtt 脚本模拟 | `emqx`             | `true`         | 跳过华为云相关 Bean      |
| 同时调试（一份硬件 + 一份模拟）    | `both`             | `true`         | 两套接入并存，业务层不感知     |

### 5.2 华为云控制台必做配置

1. **数据转发规则**：登录 IoTDA 控制台 → 规则 → 数据转发 → 新建规则，"数据源"选"设备属性 + 设备消息 + 设备状态"，"转发目标"选 HTTP 推送：
   
   ```
   URL: http://<后端外网IP>:9091/api/huawei/push
   ```
   
   `/api/huawei/push` 已在 `AuthFilter` 里加入白名单，**不**要求 Bearer token。生产应该在反代里限制源 IP 或加签名校验。

2. **设备 ID 绑定**：在 `devices` 表的 `huawei_device_id` 列填入 IoTDA 设备详情里的 "设备 ID"（形如 `5fxxxxxxxxxxxxxxxx_light001`）。否则上报会被 `HuaweiPushController` 以 `未绑定设备` 日志丢弃。

3. **IAM 子账号**：在"统一身份认证服务 IAM"创建一个子账号并加入"IoTDA Administrator"或更细粒度的权限组，把账号信息填进 `huawei.iot.iam.*`。`HuaweiIamTokenHolder` 会自动获取 + 缓存 token，临近过期前刷新。

### 6.4 联调验证清单

1. **登录**：浏览器 POST `/api/login {username:"municipal01", password:"123456", role:"municipal"}` 应返回 `{ token, role, username }`。前端 `apiLogin` 会自动把 token 存入 `sessionStorage`。

2. **设备列表**：GET `/api/devices` 返回 3 条演示设备。前端首页"设备下拉框"应能选中。

3. **WebSocket**：开发者工具 Network → WS 标签应看到 `/ws?token=...` 连接成功，每 30s 一个 `heartbeat`。

4. **模拟上报**（无真硬件）：把 `streetlight.mode` 改成 `emqx`，`mqtt.enabled=true`，用 Python 脚本：
   
   ```python
   import time, json, paho.mqtt.publish as p
   p.single("group8_streetlight/light001/data",
            json.dumps({"light": 120, "lamp": True, "ts": "2026-06-30T20:15:00+08:00"}),
            hostname="47.108.58.107", port=1883)
   ```
   
   后端日志应打印 `[MQTT-recv] topic=...`，浏览器实时刷新光照值。

5. **自动联动**：把 `light001` 的 `threshold` 设为 200，连续发两次 light=120 应自动下发 `turn_on` 并写"自动开灯"告警。

6. **离线检测**：停止上报 60s 后，30s 内必有 `alarm-new` 推到前端。

7. **真硬件**：把 `mode` 切回 `huawei`，配好 IAM 凭据 + 数据转发 URL + `devices.huawei_device_id`，按 D9_iot_cloud_oc_light 烧录设备代码即可。
